■ 슬라이드 1 — 타이틀
안녕하세요. Flutter 기반 영단어 학습 앱 '오늘의 단어장'을 개발한 OOO입니다.
망각 곡선 기반 스마트 복습 시스템이 핵심인 앱입니다.

──────────────────────────────────────

■ 슬라이드 2 — 비전
영단어 공부를 시작하지만 유지하지 못하는 이유는 세 가지입니다.
반복이 지루하고, 24시간 내 70%를 잊으며, 기존 앱은 불편합니다.
이 문제를 해결하기 위해 "매일 10분, 과학적 방법으로 누구나 꾸준히 단어를 기억할 수 있다"는 비전으로 개발했습니다.

──────────────────────────────────────

■ 슬라이드 3 — 문제 정의와 솔루션
각 문제를 기술로 해결했습니다.
망각은 SRS 알고리즘으로, 발음은 TTS로,
학습 현황은 통계 대시보드로, 단조로움은 3가지 퀴즈 모드로 해결했습니다.

──────────────────────────────────────

■ 슬라이드 4 — WBS
기획, 설계, 개발, 테스트, 배포 5단계로 진행했습니다.
설계 단계에서 아키텍처와 ADR을 문서화했고, 테스트까지 완료 후 GitHub에 배포했습니다.

[ 예상 질문 & 답변 ]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q1. 왜 Provider 대신 setState를 썼나요?
→ 이 앱은 화면 간 공유가 필요한 전역 상태가 없습니다.
   각 화면이 독립적으로 데이터를 로드하는 구조라 setState로 충분했고,
   Provider 도입은 오히려 복잡도만 높이는 오버엔지니어링이라 판단했습니다.

──────────────────────────────────────

Q2. SharedPreferences를 선택한 이유는?
→ 단어 데이터는 Word 객체를 JSON으로 직렬화해 저장하는 단순한 구조입니다.
   SQLite는 테이블 설계와 쿼리 작성이 필요해 복잡도가 높아지는 반면,
   SharedPreferences는 키-값 구조로 충분히 처리 가능했습니다.

──────────────────────────────────────

Q3. SRS 알고리즘을 직접 구현한 이유는?
→ 외부 라이브러리는 앱 규모에 비해 의존성이 무겁고 커스터마이징이 어렵습니다.
   1→3→7→14→30일 간격 로직은 단순해서 직접 구현하는 게 더 가볍고 유연했습니다.

──────────────────────────────────────

Q4. 앱의 디렉토리 구조는 어떻게 되나요?
→ lib 아래에 models, data, services, screens, widgets, theme
   총 6개 폴더로 나뉩니다.
   기능별로 명확하게 분리해 어느 파일이 어떤 역할인지 바로 파악할 수 있습니다.

──────────────────────────────────────

Q5. 빌드와 배포 과정을 설명해주세요.
→ flutter build apk --release 명령으로 릴리즈 APK를 생성합니다.
   생성된 APK에 키스토어로 디지털 서명을 하고, GitHub Releases에 업로드합니다.
   빌드는 소스코드를 기계어로 컴파일하는 과정이고,
   배포는 그 결과물을 사용자가 받을 수 있도록 공개하는 과정입니다.

──────────────────────────────────────

Q6. 오프라인에서도 작동하나요?
→ 네, 완전히 오프라인에서 동작합니다.
   단어 데이터 1500개를 앱에 내장했고,
   학습 기록도 기기의 SharedPreferences에 저장하기 때문에
   인터넷 연결 없이도 모든 기능을 사용할 수 있습니다.

──────────────────────────────────────

Q7. 테스트는 어떻게 진행했나요?
→ 단위 테스트로 SRS 알고리즘의 핵심 함수인 rateKnow, rateAgain의
   레벨 변화와 경계값을 검증했습니다.
   통합 테스트로 StorageService의 저장과 불러오기 흐름을 확인했고,
   실기기인 Galaxy S926N에서 UI 수동 테스트도 진행했습니다.

──────────────────────────────────────

Q8. Flutter를 선택한 이유는?
→ Android와 iOS를 단일 코드베이스로 개발할 수 있어 생산성이 높습니다.
   위젯 기반 UI 구조가 컴포넌트 재사용에 유리하고,
   3D 플립 애니메이션 같은 UI 구현도 자유롭게 할 수 있었습니다.

──────────────────────────────────────

Q9. 성능 최적화를 위해 어떤 노력을 했나요?
→ 플래시카드 애니메이션은 AnimationController를 직접 제어해
   불필요한 리빌드를 최소화했습니다.
   단어 목록은 ListView.builder로 화면에 보이는 항목만 렌더링하고,
   저장소 접근은 비동기 처리로 UI 블로킹을 방지했습니다.

──────────────────────────────────────

Q10. 향후 개선 계획이 있나요?
→ 단기적으로 Free Dictionary API 연동으로 발음과 예문 자동 입력을 구현합니다.
   중기적으로 flutter_local_notifications로 복습 알림을 추가하고,
   장기적으로 Firebase로 클라우드 동기화를 구현해
   여러 기기에서 학습 데이터를 공유할 수 있게 할 예정입니다.
